Shared patterns across all baskets

  1. Insurance & health records are entangled. Lassie, FirstVet, and Airvet all bundle financial products directly with health data — data is the retention/monetisation hook.
  2. "Share via simple code/link" is the recurring trust mechanism. MyTherapy's one-time doctor code, Rover's shared pet profile, BabelBark's multi-party sync — all lightweight, time-boxed sharing rather than full account access.
  3. Mobile-first is the default — even B2B-core products (PetDesk, Airvet) ship a clean, simple owner-facing app regardless of backend complexity.

Key differences

  1. Manual vs. passive data entry. Tractive logs automatically via hardware; Lassie, Felmo, MyTherapy, PetPal rely on manual/photo entry — UX must minimise this friction.
  2. Single vs. multi-species support. Most direct competitors cover dogs/cats only; PetPal's 5-species scope has fewer reference patterns.
  3. Role-based multi-user access is rare. Only BabelBark attempts true multi-stakeholder access — genuine whitespace, but little proven UX to copy.
03 — Interaction models

UX patterns

Five fundamentally different patterns for the core task: keeping pet info updated and sharing it with vets & sitters.

01

Living Shared Profile role-based

How it works
A structured profile lives on a server; owner, vet, and sitter get different permission levels (edit/add/view-only) and always see the current state.
Where used
BabelBark, shared medical records, Google Docs with permission tiers.
When suitable
Ongoing relationships with repeat collaborators (regular vet, co-owner, recurring sitter) where info changes often.
When it breaks
One-off interactions (emergency sitter, one-time vet) — granting account-based access feels heavy for a five-minute need.
02

Point-in-Time Export / Snapshot

How it works
A frozen snapshot (PDF, link, or one-time code) is generated at the moment of sharing — recipient needs no account.
Where used
MyTherapy's one-time doctor code; exporting a vaccination PDF for boarding.
When suitable
Formal, occasional handoffs where a fixed record is the actual requirement (insurance claims, legal docs).
When it breaks
When info changes frequently — the snapshot is stale the moment a new vaccine/med is added.
03

Quick-Reference Care Card

How it works
A short, scannable card auto-pulls key fields (feeding, meds, allergies, "not friendly with") into one screen for someone who's never met the pet.
Where used
Rover's sitter handbook.
When suitable
Short-term caregivers (walkers, sitters) who need fast context, not deep history, and won't edit the record.
When it breaks
When the recipient needs depth (full medical history) — a card can't hold nuance or two-way logging.
04

Conversational / Chat-Based Updates

How it works
Info exchanged via natural-language messages in a thread — "can she eat chicken today?" "yes, just no dairy."
Where used
PetDesk-style clinic messaging; WhatsApp groups with sitters/vets (today's informal default).
When suitable
Nuanced, situational judgment calls that don't map to structured fields.
When it breaks
As a system of record — info gets buried in scrollback, isn't searchable, "current state" must be reconstructed.
05

Collaborative Freeform Document / Wiki

How it works
A shared doc/database where anyone can add their own structure — sections, tables, notes — with no imposed schema.
Where used
Notion pet trackers, shared spreadsheets.
When suitable
Tech-savvy power users with custom needs, willing to build & maintain their own structure.
When it breaks
Onboarding anyone else — a vet/sitter has to learn a bespoke layout from scratch every time.

04 — Direction

Conclusions

Fit assessment of the five patterns against CLAUDE.md, plus the mechanisms worth transferring into PetPal's MVP.

Best fit

Living Shared Profile

  • Brief explicitly requires differentiated roles (owner edit / vet add medical / sitter view-only) — only this pattern natively supports it.
  • "Transferable and accessible between vets" depends on info always being current — snapshots go stale immediately.
  • It's the structural foundation everything else (reminders, records, personality notes, messaging) plugs into.
Second-best, under condition X

Quick-Reference Care Card

Condition: sharing with a one-off caregiver (last-minute sitter, single vet visit).

  • Generated from the living profile, no login required for the recipient.
  • Not a replacement for the living profile — the right "view" layered on top for this use case (à la Rover).
Doesn't fit

Freeform Wiki

  • Audience spans 16–70 incl. non-technical users — a wiki requires owners to design their own structure.
  • Vets/sitters need consistent layouts across pets/owners — relearning a bespoke doc each time fails "transferable."
  • This is the status quo (Notion) PetPal is meant to replace.

Top 3 mechanisms to transfer into MVP

01 · BabelBark's instant shareable health dashboard 02 · MyTherapy's one-time share code 03 · Rover's sitter-facing care card

1 mechanism that will not work

BabelBark's ~5,000-brand diet/nutrition database. Requires ongoing content-ops disproportionate to MVP value. A lightweight, manual "owner notes" approach (à la Rover) covers the actual JTBD — favourite treats, solid no's — without the maintenance burden or staleness risk.
